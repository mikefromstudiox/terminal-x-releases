## §2. 🔴 OWNER-VISIBLE — must ship in v2.16.28

### B1. NCF sequence provisioning seeds non-existent columns
**This is the actual Ranoza root cause.**

- **Discovered by:** silent-failure audit (panel.js:1458 + 1618)
- **Full path:**
  1. New client signs up → `web/api/signup/provision.js` calls `panel.js create_business`.
  2. `panel.js:1458` (admin) and `:1618` (self-serve) seed `ncf_sequences` rows with columns `next_number` + `max_number`.
  3. **Real schema columns** are `current_number` + `limit_number`. PostgREST silently drops unknown keys.
  4. Resulting row either has `current_number=NULL` (NOT NULL violation → row never inserts) OR doesn't insert at all silently.
  5. Web `api.ncf.next` calls `atomic_next_ncf` RPC → finds no row → returns null.
  6. CobrarModal falls back to in-memory seq from `ncfSeqs` (also empty since `sequences()` returns `[]`).
  7. Final fallback: `${prefix}${String(0+1).padStart(8, '0')}` = `B0200000001` for every ticket.
- **Root cause:** column-name drift between provisioning code and live schema. The migration that renamed columns was never propagated to the seed paths.
- **Fix (web/api/panel.js:1458 + 1618):**
  ```js
  // BEFORE
  await supabase.from('ncf_sequences').insert({ business_id, type, prefix, next_number: 1, max_number: 500, ... })
  // AFTER
  await supabase.from('ncf_sequences').insert({ supabase_id: crypto.randomUUID(), business_id, type, prefix, current_number: 0, limit_number: 500, enabled: false, active: true })
  ```
- **Verification:**
  1. Provision a sandbox business via admin create_business.
  2. Query `ncf_sequences WHERE business_id=<sandbox>`.
  3. Confirm 2-4 rows exist with `current_number=0`, `limit_number=500`, no NULLs.
  4. Add to `pre-launch-check.mjs`: every active business must have ≥1 enabled `ncf_sequence` for `B01` or `B02` or `E31`.

### B2. `tickets.create` swallows non-network errors via `tryOr`
**This is why beverage_subtotal went unnoticed for so long.**

- **Discovered by:** action audit (F1)
- **Full path:**
  1. Cashier hits Cobrar → CobrarModal → `onConfirm` → POS handlePaymentConfirm → `api.tickets.create`.
  2. `api.tickets.create` is wrapped in `tryOr(fn, fallback)`. `tryOr` catches every throw and returns the fallback.
  3. RLS denial, FK violation, NOT NULL violation, CHECK constraint failure → all caught, return `null` to caller.
  4. POS sees no error → CobrarModal flips to success → factura screen prints → cashier collects payment.
  5. Meanwhile DB has no row. Inventory not deducted. Client balance not updated.
- **Root cause:** `tryOr` is the read-side safety net (returns empty list on read failure). Using it on a critical write is wrong; should be `tryWrite` which re-throws.
- **Fix (web.js:2619):** replace outer `tryOr(async () => {...})` with explicit try/catch that ONLY catches network errors (offline) and re-throws everything else so CobrarModal can show the real error in red.
  ```js
  // BEFORE
  return await tryOr(async () => { /* … insert ticket … */ })
  // AFTER
  try { return await /* … insert ticket … */ }
  catch (err) {
    if (!navigator.onLine || /Failed to fetch|NetworkError/i.test(err?.message || '')) {
      const offlineId = await enqueueTicket(data)
      return { id: `offline-${offlineId}`, ..., offline: true }
    }
    throw err
  }
  ```
- **Verification:** force a 400 (e.g. send invalid total) → CobrarModal must show the Postgres error message in red, NOT show success. Verify with already-shipped CobrarModal success-after-DB-lands fix.

### B3. Mi Empresa double source of truth — owner edits don't reach receipts
- **Discovered by:** settings audit §2.A
- **Full path:**
  1. Owner opens Mi Empresa → edits "Nombre del negocio" → clicks Guardar.
  2. `Admin.jsx:1784-1812 saveEmpresa` writes to `businesses` table (column `name`).
  3. CobrarModal e-CF / receipt builder reads `bizSettings.biz_name` from `app_settings` KV.
  4. The two are kept in sync ONLY by a one-time `auto-heal` at migration (`electron/database.js:1516-1520`). Owner edits AFTER migration don't propagate.
  5. Result: invoice still prints the OLD name; e-CF emits the OLD name to DGII.
- **Root cause:** schema migration intended `app_settings` as the source of truth but `Admin.jsx saveEmpresa` was never updated to mirror.
- **Fix (Admin.jsx:1784-1812 saveEmpresa):** after writing `businesses`, ALSO call `api.settings.update({ biz_name, biz_rnc, biz_address, biz_city, biz_phone, biz_email, biz_website, biz_logo })`. Single transaction logically; if either side fails, surface error.
- **Verification:** edit each Mi Empresa field → save → grep `app_settings` for that business → confirm the matching `biz_*` key has the new value within 2s. Then issue a test e-CF and confirm DGII XML carries the new value.

### B4. `biz_website` silently dropped by saveEmpresa allowed-list
- **Discovered by:** settings audit §2.C
- **Full path:**
  1. v2.16.27 added `biz_website` input to Mi Empresa form.
  2. Admin.jsx writes via `api.settings.update({ biz_website })`.
  3. `web.js::saveEmpresa` (line 765-984) filters payload through an allowed-list that does NOT include `biz_website`.
  4. Key dropped before write. UI flips to "Guardado ✓" anyway.
- **Root cause:** allowed-list out of sync with whitelist. Whitelist has `biz_website`, save handler doesn't.
- **Fix:** add `biz_website` to the saveEmpresa allowed-list at line ~766.
- **Verification:** save a non-empty website → reload form → field shows the saved value. Repeat after sign-out/sign-in.

### B5. `setup_complete` written to wrong table
- **Discovered by:** settings audit §2.B
- **Full path:**
  1. FirstTimeSetup.jsx:1849 writes `setup_complete: '1'` via `api.admin.saveConfiguracion`.
  2. That goes to the legacy `configuracion` table (single-row blob, not KV).
  3. OnboardingChecklist + OnboardingWizard read `setup_complete` from `app_settings` KV.
  4. Wizard re-fires on every fresh device login because `app_settings.setup_complete` is never set.
- **Root cause:** mid-migration straddle. Half the writes went to `configuracion` blob, half to `app_settings` KV. `setup_complete` got stuck in the wrong half.
- **Fix:** change FirstTimeSetup.jsx:1849 to call `api.settings.update({ setup_complete: '1' })`. Add a one-time data migration: copy any `configuracion.setup_complete='1'` rows to `app_settings.setup_complete='1'` for businesses where the KV key is missing.
- **Verification:** complete first-time setup → sign out → sign in on a fresh browser → wizard does NOT re-fire.

### B6. Admin app_settings upsert targets a partial unique index
- **Discovered by:** silent-failure audit (panel.js:1576)
- **Full path:**
  1. Admin override (panel.js:1576) does `app_settings.upsert({ ... }, { onConflict: 'business_id,key' })`.
  2. The unique index is partial (`WHERE business_id IS NOT NULL`). PostgREST cannot use partial indexes as on_conflict targets.
  3. First save: INSERT succeeds. Subsequent saves: PostgREST rejects with 42P10 → catch silently swallows → admin thinks save worked, didn't.
- **Root cause:** memory rule "Real UNIQUE CONSTRAINT, not partial index. PostgREST rejects partial indexes" was violated when the index was created.
- **Fix:** migration to replace the partial unique index with a full `UNIQUE (business_id, key)` constraint. Code change unnecessary once migration lands.
- **Verification:** admin overrides any setting twice → second save round-trips correctly.

### B7. DGII environment flip has no post-write verify
- **Discovered by:** silent-failure audit (web.js:5182)
- **Full path:**
  1. Admin → DGII → toggle Pruebas ↔ Producción.
  2. `api.dgii.setEnvironment` updates `businesses.settings.dgii_environment`.
  3. No verification read after the write. If RLS denies (admin user is on a different business in the JWT), update no-ops silently.
  4. Next e-CF goes to the WRONG endpoint (cert↔producción mismatch) → DGII rejects → fiscal incident.
- **Root cause:** no post-write read. Same UPDATE-zero-rows class as B6.
- **Fix:** after `setEnvironment` updates, re-SELECT `businesses.settings.dgii_environment` and assert it equals the requested value. Throw on mismatch.
- **Verification:** flip env → assert read returns the new value. Force a permission denial → expect throw, not silent success.

## §3. 🟡 CASHIER-VISIBLE — ship in v2.16.28 if simple, else v2.16.29

### C1. `updateItemPrice` recompute now double-taxes (regression from earlier today)
- **Discovered by:** action audit (F2)
- **Full path:**
  1. Manager-gated price edit on a queued ticket (Queue.jsx → updateItemPrice).
  2. Recompute SELECT pulls `price, quantity` (I removed `aplica_itbis` earlier in v2.16.27 to fix the 400).
  3. Recompute logic at web.js:3437 reads `it.aplica_itbis` — now ALWAYS undefined.
  4. `if (it.aplica_itbis !== 0)` → undefined !== 0 → TRUE → ITBIS applied to ALL lines including exempt ones.
- **Root cause:** I patched the SELECT but not the consumer.
- **Fix:** since ticket_items doesn't have `aplica_itbis` column, derive from the linked service via a JOIN: `SELECT … , service:services!service_supabase_id(aplica_itbis)`. Or simpler: use the stored `it.itbis` non-zero check instead of recomputing aplica.
- **Verification:** edit price on a ticket with mixed taxable/exempt items → confirm ITBIS recomputes correctly per line.

### C2. Vehicle reservation status flip silently swallowed
- **Discovered by:** action audit (F3)
- **Full path:**
  1. Sales rep clicks "Reservar" on a vehicle in Concesionario.
  2. `dealership.reservations.create` inserts the reservation.
  3. `vehicle_inventory.status='reserved'` UPDATE wrapped in empty `try{}catch{}`.
  4. UPDATE fails (RLS, FK, anything) → silently swallowed → row says `available` → another rep can re-reserve same VIN.
- **Root cause:** empty catch.
- **Fix:** replace `catch{}` with explicit error logging + propagate to UI as a warning. Even better: do both writes in a single RPC for atomicity.
- **Verification:** force the UPDATE to fail (e.g. wrong VIN) → confirm error surfaces, reservation row is rolled back.

### C3. Returns flow has no rollback
- **Discovered by:** action audit (F4)
- **Full path:**
  1. Cashier returns 3 lines on a ticket.
  2. `notas_credito` row inserted.
  3. Per-line `inventory.adjust` calls run sequentially.
  4. Line 2 fails (e.g. inventory_item missing). Lines 1 + 3 already adjusted. notas_credito persists. Line 2 stock not restored.
  5. Customer goes home with credit; physical stock vs system stock drift.
- **Root cause:** no transactional wrapper. Sequential best-effort writes.
- **Fix:** consolidate into a single Postgres function `apply_return(ticket_id, lines)` that does all writes in one transaction; on any failure, raise + roll back.
- **Verification:** force one line's inventory adjust to fail → notas_credito should NOT exist post-failure.

### C4. Queue ticket updates have int-vs-UUID key drift
- **Discovered by:** silent-failure audit (web.js:3047 + 3832)
- **Full path:**
  1. Queue move/reorder writes `queue` row updates.
  2. Some paths use `ticket_id` (legacy int FK), some use `ticket_supabase_id` (UUID).
  3. If the row was created via the new UUID path but updated via the int path (or vice versa), `.eq()` matches 0 rows → silent no-op.
  4. Result: queue position/status appears to update locally but doesn't persist; on next refresh the move is lost.
- **Root cause:** mid-migration of the supabase_id architecture. Some queue paths weren't updated.
- **Fix:** sweep web.js for `queue.update().eq('ticket_id', …)` and convert to `.eq('ticket_supabase_id', …)`. Verify schema has both columns and the FK points to ticket_supabase_id.
- **Verification:** move a queue ticket → refresh → position persists.

## §5. Architectural debt — must address but separate work

### A1. Two sources of truth: `businesses` table vs `app_settings` KV
- This is the root of B3 + B4 + B5 + likely more.
- **Right answer:** pick ONE source. Either:
  - **Option A** — `businesses` columns are canonical. Move all reads off `app_settings.biz_*`. Update receipts, e-CF, CobrarModal to query the businesses row directly. Simpler queries, foreign keys work.
  - **Option B** — `app_settings` KV is canonical. Mi Empresa save writes ONLY to KV. The `businesses.name`/`.rnc`/etc. become DEPRECATED columns or views over KV. More flexible (arbitrary keys), harder to FK.
- Recommendation: **Option A** for fields that are universal (biz_name, biz_rnc, biz_address, etc.) + Option B for feature toggles (`feature_*`, `mgr_gate_*`).
- This is bigger than v2.16.28; ship as v2.17.0.

### A2. `tryOr` vs `tryWrite` discipline
- Currently many writes use `tryOr` because the original author wanted to "always return something safe."
- **Right rule:** ANY mutation must use `tryWrite` (re-throws). `tryOr` is reserved for reads that have a sensible empty/default fallback.
- Fix: sweep web.js for `tryOr(... insert/update/upsert/rpc ...)` patterns and convert to `tryWrite`. The action-audit's F1 finding (B2 here) is the most damaging instance; there may be others.
